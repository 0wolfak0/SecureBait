Lab Setup Architecture

How to structure a safe, isolated environment for phishing simulation training.

⚠️ Critical Requirement

The lab environment must be completely isolated from production networks. No real user data, real domains, or internet-connected systems should be involved.

High-Level Architecture

A phishing simulation lab consists of several isolated components communicating within a private virtual network. Below is the recommended architecture:

Core Components

1. Network Isolation Layer

  • Virtual Network: Use host-only or internal-only networking in your hypervisor (VirtualBox, VMware, or Proxmox)
  • No Internet Access: The simulation network must have zero connectivity to the outside internet
  • DNS Spoofing (Internal): A local DNS server resolves fictional domains (e.g., mail.training-example.local) to internal IPs
  • Firewall Rules: Block all egress traffic to prevent accidental data leakage

2. Simulation Server (GoPhish)

  • Open-source phishing simulation platform
  • Manages campaigns, email templates, and landing pages
  • Tracks user interactions (opens, clicks, submissions)
  • Runs on a dedicated VM within the isolated network

3. Mail Server (Sandboxed)

  • Local SMTP/IMAP server (e.g., hMailServer on Windows, Postfix on Linux)
  • Handles email delivery within the lab only
  • Pre-configured with dummy user accounts (e.g., trainee01@training.local)
  • No relay to external mail servers

4. Web Server (Training Pages)

  • Hosts mock phishing awareness pages
  • All pages include clear "TRAINING SIMULATION" banners
  • Served over HTTP within the isolated network (no real TLS certificates)
  • Captures form submissions for metrics (no real credential storage)

5. Trainee Workstations

  • Virtual machines simulating end-user desktops
  • Pre-configured email clients connected to the sandboxed mail server
  • Web browsers for interacting with simulation landing pages
  • Standard OS images (Windows 10/11 or Ubuntu Desktop)

6. Monitoring & Logging

  • Centralized logging of all simulation activity
  • Network traffic capture (Wireshark/tcpdump) for analysis
  • Dashboard for real-time campaign monitoring
  • All logs stored locally — no cloud services
✅ Best Practice

Document your lab setup thoroughly. Include network diagrams, IP assignments, VM configurations, and access credentials. This documentation serves as part of your academic submission.

Safety Checklist

  • Lab network has zero internet connectivity
  • All domains used are fictional (e.g., .local, .test, example.com)
  • No real user data or PII is present in the lab
  • All training pages display clear simulation banners
  • Lab access is restricted to authorized participants only
  • Written authorization from academic supervisor is obtained
  • All simulation data is destroyed after the training exercise
  • Incident response plan exists in case of accidental exposure
